### ğŸ™‹ _"I'm working on backend business logic"_

The main guide focuses heavily on this because it's where TDD works perfectly:

- **Pure business logic** â†’ Unit tests with mocked dependencies

<a id="infrastructure"></a>

### ğŸ™‹ _"I'm building technical infrastructure (adapters, repositories, etc.)"_

**`AAID` handles infrastructure differently:** This is Way 2 from Level 3:

- **Database adapters** â†’ Integration tests with real DB (managed dependency)
- **External API clients** â†’ Contract tests with mocked endpoints (unmanaged dependency)
- **Message queue handlers** â†’ Integration tests with real queue

The key difference: Instead of unit tests with mocks, you use integration/contract tests that verify technical contracts.

<a id="e2e"></a>

### ğŸ™‹ _"I need to write E2E tests (like Playwright)"_

**`AAID` covers this through acceptance testing:**

- Tests the whole system from the outside
- Mocks only external dependencies you don't control
- Keeps tests fast and deterministic
- Uses Dave Farley's Four-Layer architecture

**Different from E2E**: Acceptance tests mock third-party APIs to avoid flaky tests from external failures.

<a id="styling"></a>

### ğŸ™‹ _"I'm working on pure styling/animations"_

**`AAID` knows when NOT to use TDD:** This is Way 3 from Level 3:

For pure presentation (CSS, animations, visual design):

- Skip TDD entirely - no logic to test
- Use manual validation
- Visual regression testing if needed
- Accessibility audits

**Why?** Styling is an implementation detail. Also, often you're validating the "feel", something only humans can judge.

<a id="microservices"></a>

### ğŸ™‹ _"Will this work with microservices?"_

**Yes, `AAID` works well with microservices:**

- Each microservice uses `AAID` internally just like any regular service
- While testing:
  - During development: mock other services for fast feedback
  - Pre-deployment: uses [bidirectional contract tests](../../../appendices/appendix-e/dependencies-and-mocking.md) with real service connections
- The `AAID` workflow stays the same

<a id="scaffolding"></a>

### ğŸ™‹ _"What about project setup and scaffolding?"_

**Project scaffolding sits outside `AAID`'s workflow:**

- Running framework generators (`create-vite`, `create-next-app`, etc.)
- Installing packages and configuring build tools
- Setting up linters, formatters, test runners
- Basic config files that have no custom behavior

**How to verify?** Run it once; if it works, you're done. [Acceptance tests](../../../appendices/appendix-a/docs/aaid-acceptance-testing-workflow.md) will catch integration issues if something breaks.

**When does `AAID` apply?** When you write _custom code_:

- âœ… Business logic â†’ Use `AAID` with unit tests
- âœ… Custom infrastructure (adapters, API clients, middleware, etc.) â†’ Use `AAID` with integration tests
- âœ… Presentation/UI (styling, animations) â†’ Use `AAID` workflow, skip TDD, manual validation
- âœ… Verify system meets business requirements â†’ Use `AAID` with acceptance tests
- âŒ Scaffolding (npm install, test runner setup, config files) â†’ No `AAID`, just set up. (Acceptance test will catch issues)

See [Appendix D](../../../appendices/appendix-d/handling-technical-implementation-details.md#scaffolding-and-prerequisites) for the complete distinction.

<a id="mocking"></a>

### ğŸ™‹ _"I'm confused about what to mock in tests"_

**`AAID` has clear rules:**

- **Unit tests**: Mock everything external
- **Integration tests**: Use real managed dependencies (your DB), mock unmanaged (third-party APIs)
- **Acceptance tests**: Same as integration
- **E2E tests**: `AAID` uses Acceptance Tests

There's a complete [decision diagram](../../../appendices/appendix-e/dependencies-mocking-decision-flow.mermaid) in the appendices when you need it.
